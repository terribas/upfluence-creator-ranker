# Upfluence Creator Ranker

A data-driven web app that ranks the top 10 Instagram creators per industry using the Upfluence API and a multi-signal scoring formula.

Built as part of the Upfluence Jr Growth Engineer practical assessment.

## What It Does

- Covers **5 industries**: Beauty, Fashion, Fitness, Food, Gaming
- For each industry, fetches creator profiles via the Upfluence API and runs a two-pass scoring pipeline
- Ranks the top 10 creators by **Creator Power Score (CPS)** — a composite of reach, engagement quality, and posting activity
- Displays per-creator metrics, score pillar breakdowns, and industry-level insights for brand decision-making
- Generates an **AI brand fit analysis** per creator on demand — a 2-3 sentence streaming summary powered by GPT-4o mini, grounded in the creator's actual signals (engagement rate, reach ratio, posting cadence, growth, estimated price, and cohort rank)
- Results are cached for 24 hours (ISR) so the first visitor waits ~30 s; all subsequent visitors get an instant CDN response

## Ranking Methodology

### Step 1 — Raw Signals

Six signals are extracted from each creator profile. They are the raw inputs to the scoring formula — no ranking logic at this stage.

| Signal | Source | Notes |
|---|---|---|
| `followers` | `instagrams[0].followers` | Direct API field |
| `engagement_rate` | `instagrams[0].engagement_rate` | Null on some accounts → derived as `(avg_likes + avg_comments) / followers` from per-post data |
| `avg_reach_ratio` | Computed: mean of `post.reach / followers` across last ~9 posts | Measures how far each post travels beyond the follower base. Can exceed 1.0 (100%) when posts surface on Explore or get shared |
| `posts_per_week` | Computed: posts published in last 30 days ÷ 4 | |
| `recency_score` | Computed from the latest post timestamp | 1.0 if ≤14 days ago · 0.5 if 15–30 days · 0.0 if >30 days |
| `cross_platform_bonus` | Count of TikTok / YouTube / Twitter / Twitch / Pinterest accounts with >1 000 followers | Individual follower counts per platform are also stored and displayed in the creator detail sheet |

### Step 2 — Pillars

Signals are combined into three equal-weight pillars, each capturing a distinct dimension of creator value. Every pillar is **min-max normalized (0–1) across the full industry cohort** so scores are relative to the competition, not absolute thresholds.

| Pillar | Formula | What it measures |
|---|---|---|
| **R — Reach** | `norm( log(followers) + 0.3 × cross_platform_bonus )` | Audience scale with logarithmic diminishing returns; multi-platform presence adds a bonus |
| **Q — Quality** | `norm( engagement_rate × avg_reach_ratio )` | How actively the audience engages and how many followers actually see each post |
| **A — Activity** | `norm( posts_per_week × recency_score )` | Posting consistency; penalizes dormant accounts |

### Step 3 — Creator Power Score (CPS)

```
CPS = (R × Q × A) ^ (1/3)
```

A geometric mean of the three pillars. A creator must perform well across **all three dimensions** to rank high — a single vanity metric (e.g., 50M followers + 0% engagement) collapses the score to near zero.

### Why Geometric Mean Over Weighted Linear?

Three formulas were evaluated against live API data (see `making-of.md`). The geometric mean was chosen because:

- It **penalizes single-metric dominance**: a creator with 50M followers but ghost engagement scores near zero, not decently.
- It maps to how brands actually evaluate creators: *can they reach people? does their audience act? are they consistently active?*
- Weights are implicit and equal, which is easier to justify than hand-tuned coefficients.

**Tradeoff documented:** Normalization requires seeing the full cohort before scoring any individual creator. This forces a mandatory two-pass pipeline (fetch all → then score all) rather than a streaming approach.

## Two-Pass Pipeline

```
Pass 1 — Signal Extraction
  → Auth token
  → POST /v1/matches — search by keyword + follower range + engagement > 1% → 20 IDs
  → Fetch profiles in parallel batches of 4 (200 ms pause between batches)
  → Extract raw signals per profile (null-safe fallback from per-post data)
  → Output: flat signal record per creator — no scoring at this stage

Pass 2 — Normalization & Scoring (cohort-level, pure sync)
  → Compute min/max per signal across all creators
  → Min-max normalize each signal
  → Compute R, Q, A pillars → CPS = (R × Q × A)^(1/3)
  → Sort descending → return top 10
```

**Candidate pre-filtering:** the search query includes an `average-engagement > 1%` filter, so only creators with confirmed engagement reach the profile-fetch stage. This reduces candidates from 30 to 20 and eliminates low-quality profiles before any expensive API calls.

**Parallel batching:** profile fetches run 4 at a time with a 200 ms pause between batches. Peak burst is 4 concurrent requests — well under the Upfluence 20 req/s rate limit. Estimated first-load time: ~4–5 s (vs ~25–30 s with the original sequential approach).

Pipeline results are cached for 24 hours via Next.js `unstable_cache`.

## Scalability Decisions

### 1. Structured framework over raw HTML/CSS

The UI is built with **Next.js App Router + TypeScript + shadcn/ui** rather than generated HTML and inline styles. This was an explicit choice made at the start of the project — AI coding tools can produce working HTML/CSS quickly, but the result is a dead end: no routing, no component reuse, no type safety, no build pipeline.

With a proper framework in place:
- Adding a 6th industry requires zero new files. The dynamic route `app/industry/[slug]/page.tsx` and `generateStaticParams` pick it up automatically from a single entry in `lib/config/industries.ts`.
- Every creator card, metric badge, and modal is built from shared primitives (`shadcn/ui`). New UI features extend the system rather than copy-paste CSS.
- TypeScript enforces the data contract between pipeline stages — the compiler propagates any schema change across the codebase rather than leaving it to runtime errors.

The framework adds ~30 minutes of initial setup versus vibe-coding plain HTML. It pays back immediately the first time you need to add a feature or fix a bug.

### 2. ISR over SSR — flat API call rate regardless of traffic

The pipeline runs per-industry, not per-request. Next.js `unstable_cache` with a 24-hour TTL means Upfluence is called **at most once per industry per day**, regardless of how many visitors the app receives.

| Approach | API calls at 1k visitors/day | API calls at 100k visitors/day |
|---|---|---|
| Server-side rendering (SSR) | ~5,000 (1k × 5 industries) | ~500,000 |
| ISR + 24h cache (this app) | 5 (1 per industry) | 5 |

This matters for two reasons beyond cost: Upfluence's API has rate limits and a monthly credit budget. A traffic spike under SSR would exhaust credits within hours. Under ISR, the credit consumption is deterministic and bounded — `5 industries × ~21 calls each = ~105 credits/day` independent of traffic.

The first visitor per industry per 24h period waits ~30 s while the pipeline runs; every subsequent visitor gets an instant CDN response. For a ranking dashboard where data freshness is measured in days, not seconds, this is the correct trade-off.

### 3. Config-driven industry definitions

All industry-specific logic lives in a single file: `lib/config/industries.ts`. Each industry is one object in an array:

```ts
{
  slug: 'beauty',
  label: 'Beauty',
  emoji: '💄',
  keywords: ['beauty', 'makeup', 'skincare'],
  followerRange: { min: 10000, max: 10000000 },
  insight: '...',   // brand-facing copy rendered by InsightPanel
}
```

Adding a new industry is a **config change, not a code change** — no new routes, no new components, no new pipeline logic. The dynamic route, the navbar tab, the ISR page, and the scoring pipeline all consume this config array and handle any number of entries.

This also separates the two concerns that change at different rates: the *data model* (signals, formula, pipeline) changes rarely and deliberately; the *content* (which industries, what keywords, what insight copy) changes frequently and can be updated without touching any logic code.

## Assumptions

- **Instagram as primary platform.** The CPS is Instagram-first. TikTok, Twitter, and Twitch contribute only a cross-platform bonus to the Reach pillar.
- **50 candidates per industry.** The search step requests up to 50 creator IDs per industry. The top 10 are surfaced after scoring.
- **24-hour cache TTL.** Rankings are considered fresh for one day. Upfluence data itself updates on a similar cadence for most signals.
- **Null engagement is recoverable.** The API returns `null` for `engagement_rate`, `average_likes`, and `average_comments` on some high-follower accounts. The pipeline always falls back to computing these from per-post media data.
- **Growth rate is informational only.** `community_growth_rate` is displayed on the creator detail sheet but excluded from CPS to avoid rewarding short-term spikes over sustained quality.

## Challenges Encountered

### API credits exhausted mid-development

The Upfluence trial account operates on a monthly credit budget (reset on the 1st of each month). Each full pipeline run costs roughly 105 credits (1 search + 20 profile fetches per industry × 5 industries). In production, the 24-hour ISR cache means this runs at most 5 times per day regardless of traffic — a bounded and predictable cost. During development, however, cache clears are routine: every schema change, bug fix, or pipeline tweak requires invalidating the cache to test against fresh API data. Those repeated cold runs across multiple sessions accumulated faster than expected and exhausted the monthly allowance before the project was complete.

**Solution:** The pipeline was extended with a **disk-based fallback cache** (`/.pipeline-cache/{slug}.json`). After every successful API run, results are persisted to disk. On any subsequent failure — credits exhausted, rate limit, network error — the pipeline automatically serves the last known good data from disk instead of returning an error. This is implemented in `lib/pipeline.ts` via `readDiskCache` / `writeDiskCache`, with fallback triggered at every failure point (auth, search, profile fetch).

Since the real API data wasn't retained before credits ran out, the fallback files were seeded with synthetically generated creator profiles that match the expected schema and signal ranges. Once credits reset, a single successful pipeline run replaces the synthetic data with live results — no code change required.

**Learning:** Before starting a project that relies on a third-party API with a credit or call budget, map the full development call volume explicitly: `(calls per pipeline run) × (estimated pipeline runs during dev) × (number of environments)`. In this case, that calculation would have flagged the risk early and prompted either a lower-cost exploration strategy or a fallback mechanism from day one rather than as a reactive fix.

| Challenge | Solution / Tradeoff |
